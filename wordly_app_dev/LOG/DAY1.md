## 1. Practical Vibe Coding이란?
- 바이브 코딩이란 AI가 알아서 코드를 만들게 하는 것이다
처음에는 빠른데 첫 번째 기능, 두 번째 기능, 세 번째 기능까지는 잘 만들어지는 것처럼 느껴지나 그 이후에 진심 빡칠정도의 문제가 발생한다..
하나를 고치면 다른 기능이 깨지고, 다시 고치면 또 다른 코드가 이상해지고 미치고 팔짝 뛰어서 찾아보게 되어 한번 구현해보고자 기록한다

- Practical Vibe Coding AI AI가 프로젝트를 이해할 수 있을 만큼의 구조를 추가하는 방식이다

- agents.md 파일추가하기
    AI가 어떤 역할을 해야 하는지
    이 앱이 어떤 앱인지
    어떤 기술 스택을 쓰는지
    폴더 구조는 어떻게 되어 있는지
    스타일링 규칙은 무엇인지
    반복해서 지켜야 할 패턴은 무엇인지

강의에선 이런식으로 시작하라고 지정해준다

```
You are an expert React Native and Expo engineer.
Write clean, simple code.
Prioritize clarity over unnecessary abstraction.
```

한국어로 풀면:

너는 React Native와 Expo에 능숙한 개발자다.
코드는 깔끔하고 단순하게 작성해라.
불필요하게 복잡한 추상화보다 명확성을 우선해라.

이런 역할 정의만으로도 AI가 생성하는 코드의 성격이 달라진다고 한다


## 2. 왜 agents.md가 필요한가?

AI 도구는 매번 이전 세션의 내용을 완벽히 기억하지 못하기 때문에 agents.md가 없으면 매 프롬프트마다 이런 내용을 다시 설명해야 제대로 반영한다..

우리는 React Native를 쓴다.
Expo를 쓴다.
NativeWind를 쓴다.
Zustand를 쓴다.
폴더는 이렇게 나눈다.
이 라이브러리는 쓰지 않는다.
스타일은 이 방식으로 작성한다.

만약 이것을 잊으면 AI가 알아서 지맘대로 수정해논다

갑자기 다른 상태관리 라이브러리를 쓰거나
다른 폴더 구조를 만들거나
기존 스타일 방식과 다른 코드를 작성하거나
이미 만든 구조를 새로 갈아엎을 수 있습니다.

그 결과 프로젝트는 마치 서로 대화하지 않은 여러 명이 제각각 만든 코드같이 엉망이 된다고 한다

## 3. 좋은 프롬프트의 4단계 구조

agents.md가 프로젝트 전체의 기준이라면, 프롬프트는 “지금 당장 무엇을 만들 것인지”를 정하는 역할

프롬프트를 4부분으로 나눠서 반영한다

- 1단계: agents.md를 먼저 읽게 하기

Read the agents.md first and follow it strictly.

한국어 번역:
먼저 agents.md를 읽고, 그 내용을 엄격하게 따라주세요.
이 문장을 매번 넣는 이유는 AI가 프로젝트 규칙을 다시 문맥에 올리도록 하기 위해서입니다.


- 2단계: 한 번에 하나의 작업만 요청하기

나쁜 예시는 아래처럼 반영한 것이다

NativeWind 설정하고, 온보딩 화면 만들고, 인증도 연결해줘.
이건 세 가지 작업을 한 번에 시키면 안된다

NativeWind 설정
온보딩 화면 구현
인증 연결

이렇게 한 번에 시키면 문제가 생겼을 때 어디서 망가졌는지 알기 어려워진다

- 좋은 방식은 이렇게 나누는 것!
한가지씩 프롬프트를 넣어라

NativeWind를 Expo 프로젝트에 설정해줘

온보딩 화면을 디자인 이미지에 맞게 구현해줘.

Clerk 인증을 연결해줘.

 한 프롬프트에는 한 작업만 넣어야 합니다.


3단계: 기존 기능을 보호하는 제약 조건 넣기

AI는 “건드리지 말라”고 하지 않으면 기존 코드를 마음대로 바꿀 수 있습니다.

그래서 이런 문장이 필요합니다.

Do not change the existing screen design.
Preserve the existing UI exactly.
Keep the existing stream audio flow intact.
Do not expose any secrets in the mobile app.


기존 화면 디자인은 변경하지 마세요.
기존 UI는 그대로 유지하세요.
현재 작동 중인 Stream 오디오 흐름은 건드리지 마세요.
모바일 앱 안에 비밀 키를 노출하지 마세요.

이런 제약 조건은 이미 잘 작동하는 기능을 보호하는 안전장치입니다.

4단계: 디자인 참고 자료 첨부하기

시각적인 화면을 만들 때는 말로 설명하는 것보다 이미지를 첨부하는 편이 훨씬 좋습니다.

AI는 레이아웃 설명보다 실제 이미지를 보고 구조를 파악하는 데 더 강합니다.

영상에서는 이렇게 말합니다.

이미지 한 장이 margin 설명 세 번보다 낫다.

즉, 화면을 만들 때는:

디자인 이미지
스크린샷
참고 UI
기존 화면 캡처

같은 자료를 함께 주는 것이 좋습니다.
